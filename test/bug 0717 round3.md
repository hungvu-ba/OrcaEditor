## Bug #1 — Drag handle goes stale after Indent/Outdent (Tab/Shift+Tab)

### Description

After pressing Tab or Shift+Tab to indent/outdent a list item, the drag-and-drop handle (hover/handle machinery) isn't rebuilt against the new DOM — reported by the user simply as "Indent/outdent cần thực hiện build lại drag handler."

### Root Cause

[`media/webview/main.ts:1083-1094`](../media/webview/main.ts#L1083-L1094) (the Tab/Shift+Tab keydown handler) never calls `dragDrop.refresh()` after `commitListOpDirect`/`commitListOp` mutate the DOM, unlike `renderDocument()`, which already does exactly that at [`main.ts:352-353`](../media/webview/main.ts#L352-L353):

```ts
// main.ts:1083-1094 (current)
const outdentPlan = e.shiftKey && sel?.isCollapsed ? computeOutdent(li) : null;
const indentPlan = !e.shiftKey && sel?.isCollapsed ? computeIndent(li) : null;
if (outdentPlan) {
  commitListOpDirect(outdentPlan, dom.placeCaretAtOffsets);
} else if (indentPlan) {
  commitListOp(indentPlan, dom.placeCaretAtOffsets);
} else {
  document.execCommand(e.shiftKey ? 'outdent' : 'indent');
}
warnIfComplexTableList();
scheduleSync();
return;
```

```ts
// main.ts:352-353 (renderDocument, for contrast — already does the rebuild)
// US-17.3: drop any in-flight drag / hover handle referencing now-stale nodes.
dragDrop.refresh();
```

`refresh()` ([`drag-drop.ts:1346-1358`](../media/webview/drag-drop.ts#L1346-L1358)) is exactly the "rebuild" the report is asking for — it clears `hoveredLi`/ `hoveredBlock`, hides both floating handles, and cancels an in-flight drag. The handles are `position: fixed` elements whose coordinates are only recomputed on `mousemove`/scroll, so once Tab/Shift+Tab physically moves the `<li>` a stale handle references, it stays visually frozen at the old screen position until the mouse next moves. Neither `commitListOpDirect` (raw `Range` surgery, no `input`/`beforeinput` event at all — documented trade-off at `list-ops.ts:388-401`) nor `commitListOp` (uses `execCommand('insertHTML')`, which does fire `input`) reach a refresh today: `main.ts:601-637`'s `input` listener has no `dragDrop.refresh()` call for any `inputType` either.

### Current State Snippet

See Root Cause above — both the Tab handler (missing the call) and `renderDocument`'s existing correct pattern (what to mirror) are shown verbatim.

### Solution Direction

Add `dragDrop.refresh();` unconditionally after the indent/outdent dispatch in `main.ts`, next to the existing `warnIfComplexTableList(); scheduleSync();` at line 1092-1093 — covers all three branches (outdentPlan, indentPlan, and the legacy `execCommand` fallback), since all three mutate the DOM. **Open question, not silently decided:** the same staleness class plausibly also applies to the general `input` listener for other DOM-mutating edits — the user only reported it for indent/outdent, so per this repo's "change like surgery" principle the fix below is scoped to the Tab/Shift+Tab handler only; flag to a reviewer whether the `input` listener needs the same treatment as a separate follow-up.

### Acceptance Criteria

-   [ ]  After pressing Tab or Shift+Tab (with no subsequent mouse movement), any previously-armed drag handle no longer references a detached/stale `<li>` — either hidden, or repositioned to the correct new element.
-   [ ]  An in-flight drag (mouse already down on a handle) is safely cancelled if a Tab/Shift+Tab somehow fires mid-drag (defensive; matches `refresh()`'s existing contract for other DOM mutations).
-   [ ]  No change to drag-and-drop behavior for any case NOT involving indent/outdent.

### Test Requirements

Per the Mandatory Rule (Webview Interaction Test) — this is real hover/handle- position state invisible to a hand-built DOM snapshot. Add a Playwright case (new or extend `test/webview/list-verbs-clean-target.spec.ts`): hover an `<li>` to arm its handle (no further mouse movement), press Tab/Shift+Tab, and assert the handle no longer targets a detached node (e.g. compare the handle's bounding rect against the new element's position, or assert the old hovered `<li>` is `!document.contains(...)` while the handle reflects the change). Not coverable by `test/roundtrip/` (no live mouse/keyboard events there). Run `npx tsc --noEmit`, `npm run test:webview`.

### Isolation / Parallel Group

## Bug #2 — TC2.1b: Outdent right after Enter doesn't outdent the new item, and leaves a stray backslash

### Description

Click at the end of "Bravo" (nested under "Alpha"), press Enter (creates a new EMPTY `<li>` nested under Alpha, right after Bravo), then immediately press Shift+Tab. Expected: the new empty item outdents to become Alpha's own sibling, empty; Bravo untouched. Actual — confirmed via two user screenshots and the manual test file's current corrupted content (`test/manual/sample-hlr22-phase2-verbs.md:53-58`):

```
-   Alpha
    -   Bravo
    -   \
-   Charlie
```

The new item stayed nested as Bravo's sibling (outdent never moved it) AND serializes as a literal `\` instead of clean-empty. The user's own quoted description ("Charlie bị thụt lùi hơn so với Alpha" — Charlie ends up more indented than Alpha) does **not** reproduce in the captured evidence — Charlie stays top-level, same indent as Alpha in both the fixture and a live repro. Flagging that discrepancy explicitly rather than forcing a false match; the confirmed symptom is "no outdent happened + backslash residue," not a Charlie-specific indent defect.

### Root Cause

Two independent parts:

**(a) Wrong `<li>` targeting.** [`main.ts:1071-1072`](../media/webview/main.ts#L1071-L1072):

```ts
const anchor = sel?.anchorNode ? closestElement(sel.anchorNode) : null;
const li = anchor?.closest('li');
```

`closestElement` ([`dom-utils.ts:7-9`](../media/webview/dom-utils.ts#L7-L9)) climbs at most one level:

```ts
export function closestElement(node: Node): HTMLElement | null {
  return node instanceof HTMLElement ? node : node.parentElement;
}
```

`list-ops.ts:71-97`'s own doc comment for `normalizeToAnchor` already documents the exact failure mode this misses: *"Chrome sometimes normalizes a Range edge to the PARENT of the actual `<li>` ... especially right after something else just mutated the DOM."* That correction exists only inside `saveCaretState`/`restoreCaretState` (used AFTER a plan is already built) — `main.ts`'s initial `li` lookup has no equivalent correction. If Chromium hands back `anchorNode = Alpha's sublist <ul>` right after Enter, `closestElement(ul)` returns the `<ul>` itself, and `.closest('li')` climbs to **Alpha**, not the fresh empty item — so `computeOutdent(Alpha)` ([`list-ops.ts:202-208`](../media/webview/list-ops.ts#L202-L208)) fails its own structural guard and returns `null`:

```ts
const sublist = li.parentElement;        // top-level <ul>
const parentLi = sublist?.parentElement; // #content, not an <li>
// ...
if (!sublist || !parentLi || parentLi.tagName !== 'LI' || !grandList) return null;
```

`main.ts:1090` then falls back to `document.execCommand('outdent')` on an oddly-anchored selection, producing the observed corruption as a side effect. `computeOutdent`'s own tree-transform logic was traced line-by-line assuming it *did* receive the correct fresh `<li>` and is structurally correct for that case — the defect is in targeting, not in `computeOutdent` itself.

**(b) Backslash mechanism** — [`turndown.ts:323-332`](../media/webview/turndown.ts#L323-L332), rule `emptyListItemBr`, unchanged from the exact code already diagnosed in `bug 0717 round2.md` Bug #1:

```ts
td.addRule('emptyListItemBr', {
  filter: (node) => {
    if (node.nodeName !== 'BR') return false;
    const li = node.parentElement;
    return !!li && li.nodeName === 'LI' && li.childNodes.length === 1;
  },
  replacement: () => '',
});
```

Combined with `br: '\\'` (`turndown.ts:43`). Whatever residual `<li>` the fallback `execCommand('outdent')` leaves behind is not `childNodes.length === 1`, so it falls through to the default `br` rule. **Confirmed the already-proposed fix from `bug 0717 round2.md` is still not applied** on this branch (diffed current `turndown.ts` against that file's quoted snippet — byte identical, unfixed).

### Current State Snippet

See Root Cause (a) and (b) above — both are the full current code, verbatim.

### Solution Direction

Two independent layers:

1.  Fix `<li>` targeting at `main.ts:1071-1072` — extract a shared "resolve the `<li>` actually containing the caret, correcting for Chrome's Range-normalize-to-parent quirk" helper into `dom-utils.ts` (per CLAUDE.md's "Caret/selection placement & restore" shared-module rule), reusing the normalization logic already proven in `list-ops.ts:83-97`'s `normalizeToAnchor`. Call it before dispatching to `computeOutdent`/`computeIndent`. **Shared with Bug #4 — same call site, fix together** (see Group Targeting).
2.  Broaden `emptyListItemBr` (`turndown.ts:323-332`) per `bug 0717 round2.md` Bug #1's already-decided direction (no open decision left — that prior analysis resolved it): strip a `<br>` whenever its parent `<li>` is textually empty and every other child is another `<br>`/whitespace text, not only when `childNodes.length === 1`. **Shared with Bugs #6 and #8** (see Group Backslash) — same class of gap, different parent tags.

### Acceptance Criteria

-   [ ]  Pressing Enter to create a fresh empty nested `<li>`, then immediately Shift+Tab, outdents THAT item to become its grandparent's sibling — not its parent, not a no-op.
-   [ ]  The outdented item serializes as a clean empty bullet line, no literal `\`.
-   [ ]  Bravo's own text/nesting is unaffected in all cases.
-   [ ]  `test/bug 0717 round2.md` Bug #1's two existing fixtures still pass unchanged (regression check for the shared `turndown.ts` rule).

### Test Requirements

Per Mandatory Rules (Roundtrip + Webview Interaction Test): once the exact post-fix DOM shape is known, add a `test/roundtrip/list-ops.ts`/`lists.ts` case for the residue shape. **Critically**, `test/webview/list-verbs-clean-target.spec.ts:273-292` already has a test named *"Shift+Tab pressed right after Enter (fresh empty nested item) outdents that SAME item, not the previous one"* covering this exact scenario — but it only asserts text identity (`bravoOwnText`/`charlieOwnText`), never nesting depth or serialized markdown, so it would NOT have caught this bug as written. Strengthen it (assert `charlieLi.parentElement` is the top-level list, plus a markdown assertion) rather than only adding a new spec. Run `npx tsc --noEmit`, `npm run test:roundtrip:lists`, `npm run test:webview`.

### Isolation / Parallel Group

## Bug #5 — TC2.3b / TC2.4b / TC2.7b: selection spanning a blank line, converting to Bullet/Numbered/Task-List produces extra blank-line artifacts

**Status:** ✅ Fixed (2026-07-17, spec-bug-0717r3-5). *   \

```
*   Bravo
```

User's summary: "TC2.3b, TC2.4b, TC2.7b: lỗi, tạo thành 2 dòng trống" (creates 2 blank lines).

### Root Cause (current code — appears fixed)

[`media/webview/list-ops.ts:258-278`](../media/webview/list-ops.ts#L258-L278), `computeToList`, per `git diff -- media/webview/list-ops.ts` is currently UNCOMMITTED WIP:

```ts
export function computeToList(blocks: Element[], ordered: boolean): ListOpPlan {
  const tag = ordered ? 'ol' : 'ul';
  const itemsHtml = blocks
    .map((el) => {
      if (el.tagName === 'UL' || el.tagName === 'OL') {
        return Array.from(el.children)
          .filter((c) => c.tagName === 'LI' && (c.textContent ?? '').trim())
          .map((li) => `<li>${li.innerHTML}</li>`)
          .join('');
      }
      return (el.textContent ?? '').trim() ? `<li>${el.innerHTML}</li>` : '';
    })
    .join('');
  return {
    stableRoot: blocks[0].parentElement as Element,
    rangeStart: blocks[0],
    rangeEnd: blocks[blocks.length - 1],
    html: `<${tag}>${itemsHtml || '<li><br></li>'}</${tag}>`,
  };
}
```

The prior committed version (`2ebc1b8`) unconditionally emitted one `<li>` per selected paragraph including blank ones. The current version drops any block whose `textContent.trim()` is empty entirely. **Verified correct and robust** by feeding it through the real production pipeline (`prepareDomForSerialize`+`createTurndown`, bundled and executed against domino, same machinery `test/roundtrip/*.ts` uses — not just read): `computeToList([Alpha-p, blank-p, Bravo-p], false)` → clean `"* Alpha\n* Bravo\n"`, same result for two separate blank `<p>`s (the genuine double-Enter shape). Because the filter is `textContent`\-based (whole-block drop) rather than `childNodes.length`\-based, it's immune to the `emptyListItemBr` fragility class from `bug 0717 round2.md` regardless of how many `<br>`s/paragraphs the double-Enter leaves.

**Shared mechanism, confirmed:** ONE code path serves all three toolbar buttons. `toolbar.ts:1381-1386` (`setBulletList`'s `!current` branch), `toolbar.ts:1444-1449` (`setNumberedList`'s `!current` branch), and `toolbar.ts:1533-1546` (`toggleTaskItem`'s `!current` branch) all call the same `resolveTopLevelBlocks()` (`toolbar.ts:1491-1519`) → `computeToList()` → `commitListOp()` trio (Phase 2.7's own doc comment at `toolbar.ts:1482-1490` confirms this consolidation was deliberate). One fix covers TC2.3b, TC2.4b, AND TC2.7b simultaneously.

### Current State Snippet

See Root Cause above — the full current (uncommitted) `computeToList`.

### Solution Direction

## Bug #12 — `>` input rule on an EMPTY paragraph leaks the typed text OUTSIDE the new blockquote

### Description

**Discovered while live-verifying Bug #9's fix (Group HrCaretTrap), not from the manual TC checklist.** A genuinely separate, PRE-EXISTING defect with zero `<hr>` involvement: put the caret in any empty paragraph (`<p><br></p>`) and type `> ` then some text. Expected: a clean `<blockquote><p>text</p></blockquote>` → `> text`. Actual (live Playwright, real compiled `dist/webview/main.js`):

- Minimal repro — open an EMPTY document (`openEditor(page, '')` → `ensureTrailingParagraph` seeds `<p><br></p>`), click the paragraph, type `> Quoted line`:

```
DOM:  Quoted line<blockquote><p></p></blockquote>
save: "Quoted\n"
```

The blockquote is created but EMPTY; the typed text lands as a bare text node child of `#content` BEFORE the blockquote, and serialize drops the trailing part (`" line"` after the space-triggered rule) — net user-visible result is a stray `Quoted` paragraph plus an empty `> ` blockquote.

This is why HrCaretTrap's Bug #9 is only *half* user-visible-fixed: Layer 1 (`ensureCaretSpotBeforeHr`, `main.ts`) correctly stops the `<hr>` from being absorbed (the actual reported `> ---` corruption is gone — the `<hr>` stays a standalone thematic break), but typing a blockquote into that new empty caret-trap `<p>` still leaks text because of THIS bug.

### Root Cause

**Live-confirmed mechanism, hypothesis on the exact caret math.** The `>`-space input rule fires correctly: `applySpaceInputRule` ([`input-rules.ts:238-246`](../media/webview/input-rules.ts#L238-L246)) matches `marker === '>'`, calls `stripMarkerBeforeCaret` (`input-rules.ts:86-102`, which re-adds a `<br>` when the block goes empty), then `ctx.dom.wrapInBlockquote(block)`. The wrap itself lands the caret OUTSIDE the new inner block for an EMPTY source block.

[`media/webview/dom-utils.ts:400-432`](../media/webview/dom-utils.ts#L400-L432), `wrapInBlockquote`, caret-restore tail:

```ts
const el =
  ((parent ? findInsertedElement(parent, prevElement) : null) as HTMLElement | null) ??
  (content.lastElementChild as HTMLElement);

if (startOffset !== null && endOffset !== null) {
  placeCaretAtOffsets(el, startOffset, endOffset); // el = the <blockquote>, offsets 0/0
} else {
  placeCaretIn(el);
}
```

`findInsertedElement` returns the inserted `<blockquote>` (not its inner `<p>`). For a NON-empty block, `placeCaretAtOffsets(blockquote, n, n)` works because `locateOffset` walks the blockquote's text nodes down INTO the inner `<p>` to find character `n`. For an EMPTY block there are no text characters, so `locateOffset(blockquote, 0)` resolves to a position at/around the blockquote's own boundary (before its child) — the subsequent keystrokes then insert into `#content` beside the blockquote instead of inside it.

**Same family as Bug #8** (`<blockquote><br></blockquote>` → `> \`): both are the "empty blockquote" edge of the Phase 2.6 `wrapInBlockquote`/blockquote path.

### Current State Snippet

See Root Cause — `dom-utils.ts:400-432` verbatim (the `el` + `placeCaretAtOffsets` tail).

### Solution Direction — with a FAILED attempt already ruled out

**Do NOT re-try this (empirically verified insufficient):** retargeting the caret to the inner block, i.e. `const caretTarget = el?.firstElementChild ?? el; placeCaretAtOffsets(caretTarget, startOffset, endOffset)`, was built and run live — it did **NOT** fix the leak. The misplacement is deeper than just which element is passed; the caret still ends up outside.

**Next step (open — needs a live caret trace, not yet done):** instrument `wrapInBlockquote` to log `window.getSelection()` anchorNode/offset immediately AFTER the `placeCaretAtOffsets`/`placeCaretIn` call for the empty-block case, to see where the caret actually resolves and whether a subsequent focus/normalize step moves it. Likely fixes to evaluate once traced: for an empty source block, explicitly `placeCaretIn(innerBlock, /*selectContents*/ true)` or set a Range to `innerBlock` `firstChild`(the `<br>`) offset 0, rather than going through `locateOffset` on a zero-length blockquote. Confirm the fix on BOTH the empty case AND the non-empty case (regression), and check `toggleBlockquote`'s use of `wrapInBlockquote` is unaffected (it ignores the returned caret but shares the function).
